# ESLint Guide

This document covers the ESLint configuration for `backend/` and `mini-program/`.

## Tooling

- **ESLint version:** `^8.57.0` (the last v8 — stable flat config stays opt-in).
- **Base config:** `eslint:recommended` + a small set of stylistic rules.
- **No plugins:** v1 deliberately keeps the rule set to `eslint:recommended` only. Add `eslint-plugin-*` later when we have a clear motivation.

## Files

| Path | Purpose |
| --- | --- |
| `backend/.eslintrc.json` | Backend ESLint config (Node, ES2023, CommonJS). |
| `backend/.eslintignore`  | Backend ignores: `node_modules/`, `coverage/`, `*.min.js`. |
| `mini-program/.eslintrc.json` | Mini-program config (browser-like ES2023 + WeChat globals). |

## Scope

| Project | What is linted |
| --- | --- |
| backend | `src/`, `scripts/`, root-level `.js` (e.g. `run-tests.js`, `ecosystem.config.js`). The previous `eslint src/ tests/` script was widened to `eslint . --ext .js` so config files, scripts, and helpers also benefit. |
| mini-program | All `.js` files except `miniprogram_npm/` (generated by `miniprogram-ci` build). |

### Decision: tests/ excluded from backend lint pass

`backend/tests/` is **excluded** from lint in v1.

Rationale:
- The test files have many string literals (job descriptions, resume text, mock payloads) that intentionally mix quotes / line lengths to mirror real-world data.
- Many test files use top-level imports of helpers they exercise indirectly (e.g. `request`, `createApp` are imported but the test reaches them via setup elsewhere). Removing those imports would require re-shaping the test setup, which is out of scope for a tooling PR.
- Excluding `tests/` keeps the rule set strict on production code where it matters most.
- Tests still get checked by Node's built-in test runner + coverage gates.

If we want to lint tests later, the path is: copy `.eslintrc.json` to `backend/tests/.eslintrc.json` and relax the rules there (e.g. drop `quotes`).

## Rules summary (both projects)

Base: `eslint:recommended` (catches common bugs: unused vars, undefined globals, unreachable code, etc).

Custom rules added on top:

| Rule | Level | Why |
| --- | --- | --- |
| `no-unused-vars` | warn (ignores `_`-prefixed) | Strict hygiene but warn-level keeps existing code passing. `_` prefix opts out. |
| `no-console` | off | The codebase uses `console.*` heavily for logs. |
| `no-process-exit` | off | `lifecycle.js` calls `process.exit()` deliberately. |
| `eqeqeq` | error (smart) | Forbids `==` / `!=` except when comparing across types. |
| `no-var` | error | Only `let` / `const`. |
| `prefer-const` | error | Flag `let` that could be `const`. |
| `no-multi-spaces` | warn | Cosmetic; auto-fixable. |
| `no-trailing-spaces` | warn | Cosmetic; auto-fixable. |
| `semi` | error (always) | Force trailing semicolons. |
| `quotes` | error (single, allow templates) | Single quotes except for template literals. |
| `no-eval` | error | Security baseline. |
| `no-implied-eval` | error | Security baseline. |
| `no-empty` | error (allowEmptyCatch) | Empty `catch (err) {}` is common in WeChat handlers and admin routes; allow it but block other empty blocks. |
| `no-control-regex` | warn (backend only) | `sanitize.js` uses control chars intentionally to strip them; warn rather than block. |
| `no-dupe-keys` | warn (backend only) | The OpenAPI routes file has a duplicate path key (different HTTP methods); warn rather than block. |

`eslint:recommended` already includes: `no-undef`, `no-unreachable`, `no-dupe-keys`, `no-self-assign`, `no-prototype-builtins`, etc. Do not turn these off without a clear reason.

## How to run

```bash
# One project
cd backend
npm run lint         # check
npm run lint:fix     # auto-fix what's safe

cd ../mini-program
npm run lint
npm run lint:fix

# Both projects (from repo root)
npm run lint
npm run lint:fix
```

The `lint` script uses `eslint . --ext .js`, so it picks up all `.js` files under each project (respecting `.eslintignore`).

## CI integration

`.github/workflows/backend-ci.yml` runs `npm run lint` for both `backend/` and `mini-program/` after `npm ci` and before the test suite. A lint failure blocks the PR.

The CI step intentionally falls back to `npm install` for `mini-program` if `npm ci` fails (lockfile drift) so the lint step still runs.

## Adding a new rule

1. Pick a rule that solves a real problem you observed, not a hypothetical one.
2. Edit the relevant `.eslintrc.json`.
3. Run `npm run lint` locally — see how many existing files it flags.
4. If the new rule flags dozens of files, either:
   - ship it as `warn` first and fix in a follow-up commit, OR
   - scope it to `overrides` with a `files` glob.
5. Update this doc.

## When you disagree with a finding

- Fix the code.
- If the rule is wrong for this codebase, change the rule in the config (don't `eslint-disable` ad-hoc).
- `eslint-disable-next-line` is acceptable for very rare cases (e.g. intentionally unused params in a callback signature) but should not be sprinkled around.

## Install / version notes

ESLint is in `devDependencies`. The CI runner runs `npm ci`, so the lockfile pins the exact version locally and in CI.

If you are running lint on a fresh machine:

```bash
cd backend && npm install
cd ../mini-program && npm install
```

(Network access required. If offline, the CI step is the source of truth — local lint will fail until deps are restored.)
