# 开发日志 — 2026-07-15（Phase 9 Round 89 — Live Redis verify）

> 阶段：Phase 9 R89 — 真实 Redis 跑同样 8 个 multi-pod test
> 前置：[2026-07-15-phase9-round88.md](../devlog/2026-07-15-phase9-round88.md)

## 起点

R88 fake-redis test 8/8 pass. 但 fake 可能漏真实 Redis 边界 (e.g. EXPIRE 实际语义, LPUSH 顺序). user 答"A" → R89 live Redis test.

## 解法

`tests/sseReplayStoreLive.test.js` 用 `ioredis` 直连 `127.0.0.1:6379` (本机), 跑 8 个 test:
- Redis 不可用 → skip (CI 无 Redis 不 fail)
- 可用 → 真实 verify (catch fake/real 行为 drift)

每个 test 用 unique key prefix `test:{pid}-{ts}:` 避免并发碰撞, finally 清理.

## 8 tests

| # | Test | Verify |
|---|---|---|
| 1 | INCR 顺序唯一单调 | 50 个 id 严格 1..50 |
| 2 | LPUSH+LTRIM+EXPIRE pipeline | push 150 → size 100, TTL ≤ 60s, 头部 id=150, 尾 id=51 |
| 3 | LRANGE newest-first | 5 个 LPUSH 后 LRANGE = [5,4,3,2,1] (与 fake 一致) |
| 4 | TTL EXPIRE 真过期 | 设 1s TTL, 等 1.2s, key 消失 |
| 5 | INCR 跨 100 concurrent await | 全 unique, sorted = 1..100 |
| 6 | DEL 双清 | buf + id 两 key 都删 |
| 7 | store integration: push + size + ttl | store.size() 返回 count=2, ttl ≈ 86400, currentEventId=3 |
| 8 | store integration: since() chronological | 5 个 push, since(0)=5 个 1..5, since(3)=[4,5] |

## 关键发现

**Fake 假设全部与真实 Redis 一致**:
- LPUSH 顺序 (newest at head) ✅
- LTRIM 0 99 keep 100 ✅
- EXPIRE + TTL ✅
- INCR atomic 跨 await ✅
- since() reverse 逻辑 ✅

无 drift. R88 fake 可信.
