## 设计

### Ring buffer (in-memory)

```js
const REPLAY_BUFFER_SIZE = 100;
const replayBuffer = []; // [{id, event, data, ts}]

function bufferEvent(id, event, data, ts) {
  replayBuffer.push({ id, event, data, ts });
  if (replayBuffer.length > REPLAY_BUFFER_SIZE) {
    replayBuffer.splice(0, replayBuffer.length - REPLAY_BUFFER_SIZE);
  }
}

function getReplaySince(lastEventId) {
  const id = Number(lastEventId);
  if (!Number.isFinite(id)) return [];
  return replayBuffer.filter(e => e.id > id);
}
```

### Cap 计算
- tick 每 10s + heartbeat 每 15s = 1.67 events/s
- 100 events ≈ 60s × 1.67 ≈ **1 min**
- 改为更激进: 加大 cap 或长 cap?
- 100 events 对 dashboard 足够 (短断网 ~1min 内)
- 长断 (>1min): client 收 current snapshot 即可 (idempotent)

### Flow

```
client 断开 (network down 30s)
  ↓ (此期间 server 推 events 1, 2, 3 都丢)
client 重连 → sseConnectWithRetry → sseConnect
  ↓ req.headers['last-event-id'] = '5' (client 缓存的最新)
server on connect:
  - parse lastEventId = 5
  - getReplaySince(5) → [event 6, event 7, event 8] (假设 lastEventId=5)
  - res.write each event with id + event + data
  ↓
client: onEvent for each replayed → state 更新 → 接 4 frame
  ↓
后续正常 live stream (id 9, 10, 11...)
```

### Too-stale case

```
client 断开 5 min → buffer 已 rotate 5 次
client 重连 → lastEventId=5
  - getReplaySince(5) → 100 events ALL > 5 (buffered events 都是新的)
  - actually 是的: replay 100 events
  - 但这些 events 与 client 断时差不多, 等于 "重放近 1 分钟历史"
  - client UI 看上去像 snapshot 跳变, 但仍 idempotent
```

接受这个行为: 续传"近 1 分钟", 而不是精确 resume.
