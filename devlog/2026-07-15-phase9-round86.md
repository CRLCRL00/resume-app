# 开发日志 — 2026-07-15（Phase 9 Round 86 — Replay buffer TTL）

> 阶段：Phase 9 R86 — Replay buffer 自动过期 (rolling 24h TTL)
> 前置：[2026-07-16-phase9-round85.md](../devlog/2026-07-16-phase9-round85.md)

## 起点

R85 Redis INCR event id OK, 但 replay buffer 无 TTL — 长期累积 (虽然 cap=100, 极端情况 Redis 重启 + AOF 不完整). user 答"按优先级依次全部执行" → R86 TTL.

## 问题

- **Redis 重启**: AOF 恢复后可能保留旧事件 (几小时前), replay 时塞给新 client → stale
- **冷启动**: 服务空闲后 client 重连拿到旧 snapshot (24h+)
- **安全**: replay buffer 含用户数据 (city/salary), 长存不安全

## 解法 — Rolling 24h TTL

```js
// db/sseReplayStore.js
const REPLAY_TTL_SECONDS = 86_400; // 24h

async function push({ id, event, data, ts }) {
  await r.multi()
    .lpush(REPLAY_BUFFER_KEY, payload)
    .ltrim(REPLAY_BUFFER_KEY, 0, REPLAY_BUFFER_SIZE - 1)
    .expire(REPLAY_BUFFER_KEY, REPLAY_TTL_SECONDS)  // R86: 每 push 重置 24h
    .exec();
}
```

### 设计选择

| | 选 TTL | 选不 |
|---|---|---|
| 24h | cover 一天内的断线重连 (足够) | 旧事件自动清 (省内存) |
| Rolling | 每次 push 重置 → 活跃服务永不过期 | 冷服务 24h 后全清 |
| 1 round-trip | 与 LPUSH+LTRIM 同 pipeline | 无额外 RTT |

### 与 cap=100 关系

- **双保险**: cap 限制条数 (≤100), TTL 限制时间 (≤24h)
- 正常 traffic (1.67 evt/s): 100 events ≈ 1 min, cap 生效
- 突发: 24h 内最多保留 100 条 (cap 严格)
- 长期空闲: 24h 后整 key 清空

### Ops 可视化

```js
async function size() {
  return { count, ttlSeconds };  // R86: 同时报 TTL
}
async function ttl() { return ttlSeconds; }
```

`_buffer.size()` 现在返回 `{count, ttlSeconds}` — ops 一眼看出 buffer 健康度.

## 关键代码

```js
// 1 round-trip: LPUSH + LTRIM + EXPIRE
await r.multi()
  .lpush(REPLAY_BUFFER_KEY, payload)
  .ltrim(REPLAY_BUFFER_KEY, 0, REPLAY_BUFFER_SIZE - 1)
  .expire(REPLAY_BUFFER_KEY, REPLAY_TTL_SECONDS)
  .exec();

// ops inspect
async function size() {
  const [n, ttl] = await Promise.all([r.llen(...), r.ttl(...)]);
  return { count: Number(n) || 0, ttlSeconds: ttl >= 0 ? Number(ttl) : null };
}
```
