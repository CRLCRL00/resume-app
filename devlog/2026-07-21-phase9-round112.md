## 踩坑笔记

### 问题: T1 方案 C 数学算错 (orbitR 0.38 实际仍有 7.5px 重叠)

**原因**: plan 当时算 `142.5 - 90 = 52.5, 中心圆 60px, 差 -7.5px`, 但说"0 重叠 (视觉错位不显眼)". implementer 严格 TDD 加 **行为测试 R112 T1.5** (`Math.hypot(p.x - 375/2, p.y - 667/2) >= 60`) 后 fail, 如实暴露数学错.

**解决**: orbitR 0.38 → **0.40**, 再跑 57/57 pass.

**教训**: **行为测试 > 字符串匹配**. 第一版 R112 T1 测试用 `regex.test(fs.readFileSync(...))` 匹配源码 `0.38`, 抓不住真实几何回归. 加行为测试 (`dist >= 60`) 后数学错就无处藏身. 后续所有 R112 测试都遵循这个原则.

### 问题: commit message 数字出入 (T1 平板间距 187 vs 实际 217)

**原因**: implementer 沿用模板数字 `187px` 没重算. 实际 `307 - 90 = 217px`, 差 30px.

**解决**: coordinator 自己 `git commit --amend` 修数字, 新 SHA `5f21b46`.

**教训**: commit message 数学必须如实手算, 不能从旧模板复制. implementer 不验证 → coordinator 兜底.
