## 起点

R85 Redis INCR event id 已实现, 但没有 unit test 证明两个 pod 真的共享 counter. user 答"按优先级依次全部执行" → R88 multi-pod verify.

## 问题

如果 2 pod 各自部署:
```
pod A: id=1, 2, 3, ...
pod B: id=1, 2, 3, ...
                      ↑ 冲突! 重启后客户端 reconnect 用 Last-Event-ID=5
                      pod B 不知道这是 pod A 的 id 5
```

Redis INCR 解决但需要 verify.

## 解法 — In-process fake Redis

不引入 `ioredis-mock` dep, 写最小 fake (INCR + LPUSH/LTRIM/LRANGE/LLEN/DEL/EXPIRE/TTL), 用 require.cache swap 注入 `config/redis`.

两个 test store instance (`A`, `B`) 共享同一 fake → 模拟两个 pod 共享 Redis.

## 8 tests

| # | Test | Verify |
|---|---|---|
| 1 | 单 pod 顺序 INCR 唯一单调 | 1..50 严格递增 |
| 2 | 两 pod 共享 counter | 100 ids, Set.size=100, sorted = 1..100 |
| 3 | 两 pod push 合并 chronological | pod A push id 1-5, pod B push id 6-10, since(5) 返回 5 个 from B, since(0) 返回 10 个 A 在前 |
| 4 | TTL rolling | 每次 push 重置 24h, 1s 后 ttl 仍 ≈ 86400 |
| 5 | cap=100 跨 pod | 200 个 interleaved push, size=100, 包含 id 101-200 |
| 6 | clear() wipes both | 清空 buffer + counter, next id = 1 |
| 7 | currentEventId | ops 查当前 counter 值 |
| 8 | fallback path | Redis 抛错 → in-memory fallback |
